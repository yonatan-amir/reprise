01 Why Electron — why not C++

  • This app is I/O-bound, not CPU-bound. It watches a folder (idle), copies a file on save (disk-bound), and plays a rendered audio file (the OS does that). No real-time audio — so C++ buys zero speed. The disk and sound hardware are the bottleneck, not your code.
  • You already know ~90% of it. Electron = Node + Chromium. You have JS/TS, React, Node. The only new bits — main vs renderer, IPC, packaging — are about a day, learned while building.
  • Cross-platform is free. Win + Mac from one codebase. A C++ GUI + hand-built React bridge is weeks of pain for no benefit.
  • C ≠ C++. Comfort with C (42) doesn't transfer much, and "almost done with a course" ≠ "can ship a cross-platform C++ GUI+audio app." Multi-month gap, zero payoff here.
  • Where C++ does belong: the long-term north star — patchbay's in-DAW plugin (real-time audio genuinely needs C++). Keep learning it on the side for that. Not for Reprise.

02 v1 scope

Ships in v1

  • Pick folder(s) to watch (any DAW, both OSes)
  • Auto-snapshot a version on project-file change (debounced)
  • Attach an audio bounce + a note to a version
  • Version timeline per project
  • A/B audio player — two versions, level-matched, instant toggle
  • Restore a version

NOT v1 — later / the wall

  • Cloud sync, accounts, multi-device → v2
  • Plugin-chain capture & reuse → patchbay
  • Cross-DAW conversion → patchbay (boss level)
  • Collaboration, payments, AI features
  • Auto-rendering bounces (impossible outside the DAW)

03 Architecture — keeps the long road open

src/core/Pure TS, no Electron/React imports: versioning, hashing/dedup, data model. This decoupling lets you add cloud or swap the shell later without a rewrite.
src/main/Electron main process: file watching (chokidar), filesystem, SQLite, IPC. Thin — calls into core.
src/renderer/React UI only. Talks to main via IPC.

04 Data model

Projectid · name · watchedPath
Versionid · projectId · createdAt · storedFilePath · bouncePath? · note? · fileHash
